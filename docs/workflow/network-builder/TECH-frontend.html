Network Builder — client architecture

The board is a small vanilla-JS state machine: state in, board out. render = f(state, estimate(state)). This section owns the view — palette, fan-out canvas, five-metric panel, configurators, guardrail bar — the one-affordance / three-input place model, and how it mounts into tool.html and adds a Value column to choose.html. It consumes the estimate engine (defined in §5) that the Backend section implements.

1Component tree

Modules, not classes — each is a render function plus its wiring. The tree mirrors the DESIGN prototype's DOM (#path / #edges / #estimate / .china) so the approved markup evolves in place rather than restarting.

2State shape & the render loop

The entire board is derived from one small object. jump carries region/plan so multi-region providers resolve to a concrete priced variant; exits is an array (fan-out, duplicates allowed — the "1 BWH → 2 AaITR" case); traffic exists only on metered exits.

// the single source of truth — everything renders from this
interface BuilderState {
  geo: 'CN' | 'ROW' | null;            // detectGeo() result; null = unknown / lookup failed
  gatingOn: boolean;                 // China guardrail; auto from geo==='CN', user-overridable
  jump: { provider: string; region?: string; plan?: string } | null;
  exits: Array<{ provider: string; traffic?: { pkg: 100|300|500|1000; cycle: 1|3|6|12|24 } }>;
}

// render is a pure projection of state; no other source drives the DOM
function render() {
  const est = estimate(S, roster);        // §5 — pure, synchronous
  paint('#path',     Canvas(S, est));   // fan-out + connectors from est.perEdge
  paint('#canvasNotice', Notice(S));
  paint('#edges',    EdgeList(est));
  paint('#estimate', EstimatePanel(est));
  syncGuardrail(S);                       // toggle + geo stamp
}
▸ One improvement over the prototype: delegate, don't re-wire
The DESIGN prototype re-binds listeners after every innerHTML swap. Production attaches one delegated listener per stable container (#board for click/keydown, the palette for dragstart, slots for drag/drop) that reads data-* off the event target. Re-rendering then only rewrites HTML — no re-binding, no leaked handlers. After a place(), focus moves to the just-placed node (or the AddExitTarget for an appended exit) so keyboard flow is unbroken.

Actions are the only mutators — each mutates S, then calls render() and announce():

4Three inputs, one place() — the a11y contract

Each palette card is a native <button> that is also draggable. Button semantics are the baseline; drag is progressive enhancement. All three inputs converge on one guarded place(provider, role); drop targets accept only role-compatible boxes; exits append.

✓ Accessibility contract (non-negotiable)
Palette card = native <button draggable> with a descriptive aria-label ("Add BandwagonHost as jump host"); all placement funnels through one role-checked place(); empty slots are role="group"/role="button" tabindex="0" accepting only compatible boxes (reject flash + announce on mismatch); filled slots take Delete/Backspace and expose a remove <button> "✕"; a polite aria-live region narrates every place / append / remove / toggle / package change; focus moves to the just-placed node; all targets ≥ 40px with a 2px :focus-visible ring; drag ghosts and slot-highlight transitions collapse under prefers-reduced-motion.

5The estimate engine interface — assumed · reconcile with Backend

The view calls one pure, synchronous function. This is the exact contract the Frontend assumes; the Backend section must implement estimate (and the roster + detectGeo) to these shapes. No promises inside estimate — geo is the only async, resolved once at boot.

// ── the single call the view makes, every render ──────────────
function estimate(state: BuilderState, roster: Roster): Estimate

interface Estimate {
  price: {
    monthlyUSD: number;      // jump counted ×1 + Σ exits; metered folded to effective/mo; refs excluded
    native?: string;        // optional "¥149 + …" native-currency echo
    lines:   string[];      // one human breakdown line per counted hop
    metered: string[];      // fold-in math per metered exit; PM-pending → carries a ⚑ph marker
    refs:    string[];      // reference/priced-only hops, not summed
    hasRef:  boolean;
  };
  latency:   { lo: number; hi: number; illustrative: boolean } | null;  // range across exits
  io:        { iops: number; capped: boolean } | null;   // min across exits (bottleneck); null = not benchmarked
  bandwidth: { mbps: number; pending: boolean } | null;  // min across ALL hops; pending → ⚑ph
  route: {
    kind: 'datacenter' | 'residential';
    label: string;          // e.g. "2× AT&T (#1)" or joined per-exit ISPs
    purity: 'high'|'med'|'low'|'mixed' | null;
    exitIPs: number;         // N separate egress paths (0 = single-host)
    rankPending: boolean;    // ISP #rank still PM-pending → ⚑ph
    benchmarked: boolean;
  } | null;
  perEdge: Edge[];             // render-ordered: [entry, link×N]  OR  [entry, tail] for single-host
}

interface Edge {
  kind: 'entry' | 'link' | 'tail';    // entry: you→jump · link: jump→exit · tail: jump→internet
  from: string; to: string;             // display names, ready to print
  gia?: boolean;                     // entry only — is the entry CN2 GIA?
  gated?: boolean;                   // entry only — state.gatingOn && !gia  ⇒ view paints RED
  latency: { lo: number; hi: number } | null;
  bandwidthMbps: number | null;
  io: { iops: number; capped: boolean } | null;
  route: { isp: string; rank: number; purity: string; rankPending: boolean } | null;
  benchmarked: boolean;             // false → "not benchmarked" (reference hop)
  exitIndex: number | null;         // 0-based; egress-IP label = exitIndex + 1
}

// ── geo boot (async, once) — the only IO on the whole surface ──
function detectGeo(): Promise<{ country: string | null; gating: boolean }>
// resolves fast; on failure → { country:null, gating:false } and the toggle stays visible
⚑ Frontend assumptions the Backend section must confirm (so we match)
1 · estimate is pure & synchronous over (state, roster) — no fetch, no await, safe to call every render. 2 · The guardrail decision lives in the engine (edge.gated), single-sourced; the view only reads it and paints. 3 · Metered price = pkgBase(pkg) × (1 − cycleDiscount); CYCLES ships with the roster; PM-pending package prices set a pending flag so the panel stamps ⚑ph. 4 · Currency (CAD/CNY → USD at snapshot rates) is normalized in the roster/price layer; estimate returns USD in monthlyUSD, native optional. 5 · Multi-region providers resolve state.jump.{region,plan} → a concrete row; if omitted, the provider's canonical benchmarked variant. 6 · perEdge comes back already in render order; exitIPs === state.exits.length. 7 · The roster is the shared provider model — same records feed the Palette (view) and estimate (engine); neither hand-maintains a second list.

7choose.html — the Value column

The fleet table in §04 ("our real fleet, every axis") gains a derived Value = Overall ÷ $/mo column, in points-per-dollar. It reuses .perf-ref tints so it reads as native. Two design decisions keep it honest and robust:

  1. Values are pre-rendered in static HTML — derived, not fetched — so they're correct with zero JS (the site's honesty + no-backend principle). The Value column is just two new cells per row (Price $/mo, Value pt/$), computed from the roster price and the Overall already in the row.
  2. Sort + best-value highlight are progressive enhancement. A ~30-line script marks aria-sort on the Value header, reorders <tbody> rows on click/Enter (default: Value descending), and tints the max-value row .best with a ★ best value badge. Without JS the column is static but complete; the best row can also be statically pre-tinted.

⚑ Reconciliation flag — which price pairs with each row
The DESIGN mock shows "BandwagonHost 2.77 ★best" by pairing perf 83 (the 4C·4GB / $56.99 box) with $30 (the 3C·2GB box). With honest per-row pricing those are two separate fleet rows and the best value is actually BWH 3C·2GB at 2.50 (or SolaDrive 2.20 among single-listed boxes). The frontend computes ★ best value as max(value) over whatever price is attached to each row — the PM/Backend must confirm the canonical (region, plan → price) pairing so the Value column and the builder's price agree. 9Open questions & assumptions to reconcile

  1. engine Confirm the §5 estimate / Edge shapes — especially that the guardrail (edge.gated) and currency normalization live in the engine, not the view.
  2. data Canonical (region, plan → price) pairing per multi-region provider (BWH/DMIT/VMISS), so the Choose Value column and the builder agree and the ★ best value is truthful (see §7 flag).
  3. decision detectGeo() provider + fallback copy — which client geo-IP endpoint, and the toggle label when it fails (gating defaults off).
  4. price Traffic-package base prices (VirVM 100/300/500/1000 GB + base) and whether cycle discounts are provider-wide or per-package — until then the panel stamps ⚑ph.
  5. ux Max-exits cap for the fan-out (DESIGN left it unbounded, wraps cleanly) — confirm a sensible UI limit or keep unbounded.
  6. choose Whether the Choose Value column is static-pre-tinted best row + JS sort (recommended), or fully JS-driven — confirm the no-JS baseline requirement.
